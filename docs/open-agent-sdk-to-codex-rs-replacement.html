open-agent-sdk 替换为 codex-rs 设计文档

本文定义 FrogClawClient 将当前内嵌 open-agent-sdk Agent 实现替换为本地 E:\frogclaw\codex\codex-rs 的方案。替换目标不是再做双后端，而是让 Codex 成为项目唯一 Agent runtime， UI 和业务层只面对 FrogClawClient 自己的稳定事件与命令接口。

目标项目E:\frogclaw\frogclawclient
Codex 源码E:\frogclaw\codex\codex-rs
当前 SDKsrc-tauri/crates/open-agent-sdk
文档日期2026-05-03

1. 结论

采用 Codex app-server / protocol adapter 作为主方案。 FrogClawClient 不应直接把大量 codex-rs 内部 crate 类型暴露给 UI，也不应长期解析 CLI 文本输出。 推荐在 Tauri 后端增加 CodexRuntime 适配层，使用 Codex 的 app-server、client、protocol、core 能力，把 Codex 事件转换为现有 Agent UI 可消费的 FrogClaw 事件。

open-agent-sdk 的代码和 submodule 最终删除；frogclaw-agent 继续保留，但职责从 “包装 open-agent-sdk” 改为 “包装 codex-rs runtime 并输出 FrogClaw 事件”。这能最大限度保留当前前端 Agent UI、 数据库会话、权限弹窗、工具卡片和取消逻辑。

3. 替换目标

  1. 删除运行时对 open_agent_sdk 的所有 Rust import。
  2. 保留现有 Tauri command 名称，首期仍使用 agent_query、agent_cancel、权限响应等入口，降低前端改动量。
  3. 保留现有 Tauri event 名称，例如 agent-stream-text、agent-tool-start、agent-permission-request、agent-done。
  4. 会话状态从 sdk_context_json 迁移到 codex_session_json 或 runtime_context_json。
  5. 模型、provider、代理、登录状态、工作目录、权限模式由 FrogClaw 设置层统一生成 Codex runtime config。
  6. 文件写入、shell、网络、越权目录访问全部走 Codex sandbox / approval 语义，并映射到 FrogClaw 的权限 UI。
关键约束 Codex 是完整 coding agent runtime，不是 open-agent-sdk 这种轻量 loop 库。替换时不要试图逐个复刻 SDKMessage、provider bridge、tool trait；应以 Codex 的 session、turn、event、approval 为中心重新适配。

5. Codex 接入方式

方式 定位 结论
app-server / client / protocol 以结构化协议驱动 Codex，天然支持事件、session、approval、resume。 主方案。适合桌面 UI 长期集成。
CLI 子进程 通过 codex 命令快速验证。 只用于 PoC 或兜底，不作为最终架构。
直接调用 core 内部 API 把 Codex core 当 library 嵌入。 谨慎使用。只在 app-server 协议缺少能力时局部引入。
复制 Codex 代码到 FrogClawClient 把 Codex crate vendor 到当前仓库。 不推荐。后续同步成本高，容易产生分叉。

推荐依赖策略

第一阶段不把 Codex workspace 直接编进 Tauri。FrogClaw 以 sidecar 方式启动 codex-app-server --listen stdio://，通过换行分隔 JSON-RPC 通信。这样可以先避免把 E:\frogclaw\codex\codex-rs 的大量 workspace 依赖拉入当前工程。 

codex-app-server.exe 查找顺序：
1. FROG_CODEX_RS/target/release 或 target/debug
2. E:/frogclaw/codex/codex-rs/target/release 或 target/debug
3. PATH、用户 npm/bin、本地工具目录

发布前应把 codex-app-server 作为 Tauri sidecar 随包发布，或用构建脚本从内部 mirror 编译。 第二阶段再评估是否引入 codex-app-server-client 的 in-process client；只有当 sidecar 协议无法满足 权限、resume 或性能要求时，才把 Codex crates 作为 path/git dependency 引入。

6. Rust 模块设计

建议在 src-tauri/crates/agent/src 下重组模块：

src-tauri/crates/agent/src/
├── lib.rs
├── runtime/
│   ├── mod.rs
│   ├── codex.rs
│   ├── config.rs
│   ├── events.rs
│   └── session.rs
├── permission.rs
└── security.rs

核心 trait

#[async_trait::async_trait]
pub trait AgentRuntime: Send + Sync {
    async fn start_or_resume(&self, request: RuntimeStartRequest) -> AgentResult<RuntimeSession>;
    async fn send_turn(&self, session: RuntimeSession, turn: UserTurn) -> AgentResult<RuntimeEventStream>;
    async fn cancel(&self, session_id: &str) -> AgentResult<()>;
    async fn approve(&self, request_id: &str, decision: ApprovalDecision) -> AgentResult<()>;
}

CodexRuntime 职责

  • 创建或连接 Codex app-server。
  • 把 FrogClaw 会话 ID 映射为 Codex thread/session ID。
  • 调用 CodexConfigResolver 为当前会话动态生成 config.toml 和运行时环境变量。
  • 把用户 prompt、附件、cwd、权限模式、模型配置转换为 Codex turn request。
  • 监听 Codex protocol event stream。
  • 把 Codex approval 请求挂起到 FrogClaw 权限 UI。
  • 把 Codex 最终上下文写回数据库。

7. 动态配置生成与 app-server 启动

FrogClawClient 不应要求用户手工维护 ~/.codex/config.toml。替换后由 FrogClaw 设置层动态生成 Codex 专用配置目录，并在启动 Codex app-server 前传入 CODEX_HOME 和 provider API key 环境变量。

配置目录

项目 路径 说明
Codex home ~/.frogclaw/codex/ FrogClaw 管理的 Codex 专用 home，启动 app-server 时设置为 CODEX_HOME。
Codex config ~/.frogclaw/codex/config.toml 由 FrogClaw 每次运行前按当前设置原子写入，用户不需要手工编辑。
Codex sqlite/log/state ~/.frogclaw/codex/state/、~/.frogclaw/codex/log/ 内部状态，不能放到 Documents root。
Provider API key 进程环境变量 不写入 config.toml，用 env_key 间接引用。

生成的 config.toml 模板

model = "gpt-4.1"
model_provider = "frog-provider"
sandbox_mode = "workspace-write"
approval_policy = "on-request"
sqlite_home = "C:/Users/<user>/.frogclaw/codex/state"
log_dir = "C:/Users/<user>/.frogclaw/codex/log"

[model_providers.frog-provider]
name = "Frog Provider"
base_url = "https://api.example.com/v1"
env_key = "FROG_CODEX_API_KEY"
wire_api = "responses"
requires_openai_auth = false

base_url 必须来自 FrogClaw 的 provider 设置，并写到 OpenAI-compatible 根路径，例如 https://api.example.com/v1。当前 Codex provider 配置使用 Responses API， 因此第三方服务需要支持 /v1/responses；只支持 /v1/chat/completions 的服务需要额外 proxy/adapter。

API key 注入

首选方式是 env_key，例如配置中写 env_key = "FROG_CODEX_API_KEY"， Tauri 后端启动 app-server 前把用户保存的 key 注入 app-server 进程环境： 

CODEX_HOME=C:\Users\<user>\.frogclaw\codex
FROG_CODEX_API_KEY=<decrypted key from frogclaw.db>
不要默认使用 experimental_bearer_token。 experimental_bearer_token 会把 key 写进 config.toml，只允许用于调试或受控测试。 产品实现必须优先从 FrogClaw 加密存储读取 key，并通过环境变量传给 app-server。

app-server 生命周期

模式 行为 建议
sidecar app-server FrogClaw 启动随包发布的 app-server 子进程，通过 stdio JSON-RPC 通信。 第一阶段推荐，依赖隔离清晰，便于验证。
in-process app-server 把 app-server/runtime crate 编入 Tauri 后端，同进程运行。 第二阶段评估，体验更好但依赖冲突和崩溃隔离风险更高。
用户系统 codex CLI 只作为调试或兜底 runtime，不作为本替换方案主路径。 不能替代 app-server 的结构化事件和权限集成。

8. 事件映射

FrogClaw 事件 Codex 来源 说明
agent-message-id assistant message / turn item id 用于前端绑定消息 ID 和增量文本。
agent-stream-text assistant text delta 保持当前聊天窗口流式输出。
agent-stream-thinking reasoning / thinking summary delta 只展示 Codex 协议允许展示的推理摘要，不暴露不可展示内容。
agent-tool-start tool call started shell、apply_patch、file read/write、MCP 等都映射为工具卡片。
agent-tool-result tool call completed 输出要截断和脱敏，避免超大 stdout 卡死 UI。
agent-permission-request approval request / sandbox escalation 保留当前权限弹窗，但文案改为 Codex 的命令、路径、网络、沙箱上下文。
agent-status turn state / progress event 用于展示 running、waiting_permission、compacting、resuming 等状态。
agent-done turn completed 写入 token、费用、最终消息、上下文快照。
agent-error runtime error / protocol error 记录到 DB，并给 UI 可理解的错误摘要。

9. 数据库迁移

现有 agent_sessions 里有 sdk_context_json 和 sdk_context_backup_json，替换后建议新增字段：

ALTER TABLE agent_sessions ADD COLUMN runtime_kind TEXT NOT NULL DEFAULT 'codex';
ALTER TABLE agent_sessions ADD COLUMN codex_session_id TEXT NULL;
ALTER TABLE agent_sessions ADD COLUMN codex_thread_id TEXT NULL;
ALTER TABLE agent_sessions ADD COLUMN runtime_context_json TEXT NULL;
ALTER TABLE agent_sessions ADD COLUMN runtime_context_backup_json TEXT NULL;
ALTER TABLE agent_sessions ADD COLUMN runtime_status TEXT NOT NULL DEFAULT 'idle';
ALTER TABLE agent_sessions ADD COLUMN runtime_error TEXT NULL;

兼容策略：

  1. 迁移时不删除旧字段，避免旧版本回滚失败。
  2. 新会话只写 runtime_context_json。
  3. 旧 sdk_context_json 不尝试强转成 Codex 上下文；首次打开旧会话时创建新的 Codex session，并在 UI 标记为“已切换到 Codex runtime”。
  4. 确认无回滚需求后，再删除 SDK 字段和相关命令名里的 sdk 语义。

10. 设置、存储与路径

替换实现必须遵守 FrogClawClient 双根目录策略：

内容 路径 规则
Codex session metadata、索引、内部缓存 ~/.frogclaw/ 下的 runtime 子目录 属于应用内部状态，不放进 Documents。
用户上传图片、文件、备份 ~/Documents/frogclaw/ 数据库继续保存相对 documents root 的路径。
Codex 工作目录 用户选择的项目目录 必须在会话级固定，权限策略围绕该 cwd 生成。
Codex 配置 ~/.frogclaw/codex/config.toml 由 FrogClaw 设置动态生成，启动 app-server 时通过 CODEX_HOME 指向该目录。

11. 权限模型

FrogClaw 权限层应成为 UI 协议，Codex 权限层应成为执行协议。两者之间做明确映射：

FrogClaw 权限模式 Codex 映射 行为
read_only 只读 sandbox，禁用写文件和危险 shell 适合代码问答、review、解释。
workspace_write 允许 cwd 内写入，越界需要 approval 默认 coding agent 模式。
approval_required shell、网络、越权路径、破坏性操作都请求 approval 适合不信任项目或用户想逐步确认。
full_access 降低审批频率，但仍记录所有工具调用 只能由用户显式选择，不作为默认值。
安全边界 不允许 Codex 默认读写 ~/.frogclaw/ 中的数据库、master key、SSL key 或 vector DB。 如确需访问，必须由 FrogClaw 暴露受控工具，且经用户确认。

12. 迁移步骤

阶段 0：PoC 验证

  1. 确认 E:\frogclaw\codex\codex-rs 可独立构建。
  2. 在 frogclaw-agent 新增最小 CodexRuntime，只跑通一轮 prompt 并返回文本。

  4. 验证由 FrogClaw 生成 ~/.frogclaw/codex/config.toml 后，app-server 能读取非官方 base_url 和环境变量 key。

  1. 把 agent_query 内直接创建 open_agent_sdk::Agent 的逻辑迁入旧 adapter。

  3. 实现 CodexEventMapper，输出现有 Tauri event。

  1. 把 open_agent_sdk::CancellationToken 替换为 FrogClaw 自有 cancellation handle。

  3. 接入 Codex approval response，验证等待权限、同意、拒绝、取消。

阶段 3：会话与上下文

  1. 新增数据库字段并写迁移。
  2. 实现 Codex session/thread ID 持久化。
  3. 实现 resume、backup、clear context、restore context 的 Codex 版本。

阶段 4：移除 open-agent-sdk

  1. 删除 src-tauri/crates/open-agent-sdk submodule。
  2. 删除 .gitmodules 中对应条目。
  3. 删除所有 open_agent_sdk import 和 dependency。
  4. 更新 skills 命令和 UI 文案。
  5. 执行完整构建、关键路径手测和回归测试。

13. 验收标准

  • rg "open_agent_sdk|open-agent-sdk" src-tauri 不再命中运行时代码和 Cargo 依赖。
  • src-tauri/crates/open-agent-sdk 不再是 workspace 依赖，也不再是 git submodule。
  • FrogClaw 能根据设置动态生成 ~/.frogclaw/codex/config.toml，并通过 CODEX_HOME 启动 app-server。
  • API key 不落入 config.toml；app-server 通过 env_key 对应的环境变量读取 key。
  • 非官方 base_url 可用，且对不支持 Responses API 的 provider 给出明确错误或 adapter 提示。
  • Agent 模式可完成：普通问答、代码修改、命令执行、权限请求、拒绝权限、取消运行、继续会话。
  • 旧会话不会导致应用崩溃；旧 SDK 上下文字段可被忽略或显示迁移提示。
  • 所有附件路径仍按 Documents root 相对路径保存。
  • Codex 不会默认访问 FrogClaw config home 中的敏感文件。
  • Windows 路径、空格路径、中文路径的 cwd 和附件可正常传递给 Codex。

14. 风险与处理

风险 影响 处理方式
Codex workspace 依赖多，直接 path dependency 可能引入版本冲突。 构建失败或二进制体积增长。 PoC 阶段先只引入 protocol/client 层；必要时用 app-server 子进程隔离依赖。
旧 sdk_context_json 无法转换为 Codex session。 旧 Agent 会话不能无缝续跑。 保留历史消息，但新建 Codex runtime 上下文；UI 给出一次性提示。
Codex approval 语义和现有权限 UI 不完全一致。 用户可能看不到足够上下文。 权限 payload 增加 command、cwd、paths、sandbox、reason 字段。
CLI fallback 的输出解析不稳定。 工具卡片和流式消息错乱。 只把 CLI fallback 用于开发验证，正式实现使用结构化协议。
Codex 默认目录和 FrogClaw 双根目录冲突。 状态文件散落或敏感文件暴露。 所有 runtime state 都由 FrogClaw 显式指定到 ~/.frogclaw/ 下。
动态生成的 config.toml 和 app-server 已加载配置不一致。 用户修改设置后，app-server 仍使用旧 provider 或旧 key。 给配置生成结果计算 hash；hash 变化时重启 app-server 或调用 app-server 的配置刷新能力。
把 API key 写入 config.toml 或日志。 密钥泄露。 只用 env_key；日志中屏蔽 key、Authorization header 和 provider token。
